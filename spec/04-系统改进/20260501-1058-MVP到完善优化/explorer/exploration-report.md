# 探索报告：从 MVP 到场景化产品完善

## 1. 结论

CNVN 当前已经不是从零开始的 Demo，而是一个有核心业务骨架的 MVP：认证、教师档案、教师搜索、可用时段、预约、课堂消息、评价、钱包、支付订单、托管、结算快照、出款查询等模块都已经存在。

本轮不能直接按模块做“大优化”，否则会把用户体验、资金正确性、教师供给、课堂体验、运营风控混成一个不可验收的工程包。更合适的主线是先整理用户使用场景，再按场景创建独立 Spec，每个 Spec 单独定义用户目标、P0 闭环、异常路径和验收标准。

## 2. 产品定位

CNVN 是面向越南市场的中文学习双边撮合平台，连接越南学员和中文教师。产品差异点不是泛在线教育，而是：

- 越南学习者可以用越南语辅助学习中文。
- 教师供给可以来自在华越南留学生、中文导游、中文母语者和兼职自由教师。
- 平台需要承担预约撮合、在线课堂、支付托管、评价信誉、教师结算和争议处理。

## 3. 当前已覆盖的用户路径

| 场景 | 已有页面/API | 当前状态 |
|---|---|---|
| 访客发现平台 | `/`, `/teachers`, `/teachers/:id` | 首页和教师搜索已接真实 API，但本地化和筛选还弱 |
| 用户注册登录 | `/register`, `/login`, `/auth/*` | 基础可用，角色切换和教师开通已存在 |
| 学员找老师 | `/teachers`, `/teachers/:id`, `GET /teachers` | 搜索、排序、详情、评价、可用时段已存在 |
| 学员预约 | `POST /lessons` | 后端存在，前端详情页预约按钮尚未接入 |
| 教师确认/授课 | `PATCH /lessons/{id}/confirm/start/end` | 后端存在，教师 dashboard 未接状态动作 |
| 学员课堂 | `/classroom/:id`, WS 消息 | 文本消息可用，视频/资料仍是占位 |
| 课后评价 | `POST /reviews` | 后端存在，学员 dashboard 未接提交表单 |
| 钱包支付 | `/wallet`, `/payments/orders/:id`, `/wallet/topup` | 支付前端是 spike，真实支付/环境边界不足 |
| 托管结算 | PaymentOrder, SettlementSnapshot, PayoutOrder | 后端骨架存在，但 24h 内取消和评价后佣金统计有资金一致性风险 |
| 教师出款 | `/payouts`, `GET /payouts/me` | 查询可用，缺教师侧解释和运营处理 |
| 运营风控 | 无完整后台 | 缺订单检索、争议处理、人工退款、KYC 审核 |

## 4. 关键缺口

### 4.1 单课交易闭环没有被前端串起来

学员能看教师，但不能从详情页完成预约；教师能看到课程，但不能确认、开始、结束；学员能看到已完成课，但不能评价。后端状态机已经存在，产品化的第一步是把这些状态动作接到前端。

### 4.2 支付托管存在 P0 一致性缺陷

`<24h` 取消课程时，后端会把课程置为 `cancelled`，但没有写入 `PaymentOrder.held_until`。争议期 watcher 只扫描 `held_until` 到期的 held 订单，因此该订单可能永久托管，教师无法结算。

另一个缺陷是评价会把课程状态从 `completed` 改为 `reviewed`，但佣金阶梯统计只统计 `completed`，可能导致评价早于 release 时影响结算比例。

### 4.3 教师侧供给闭环不足

教师入驻、档案、可用时段、课程动作、收入解释、出款列表是一个完整工作台场景。当前后端能力分散存在，前端工作台只展示一部分，缺少可操作性。

### 4.4 运营后台缺位

一旦有真实交易，就必须处理异常订单、争议、退款、教师审核、支付回调失败和账本对账。当前模型能支持一部分，但没有运营人员可使用的闭环。

### 4.5 验证体系按模块而不是按场景

现有 pytest 覆盖认证、教师、预约、评价、课堂消息；支付 v2 主要依赖历史 E2E 脚本，没有进入常规 pytest/CI。产品化应该按真实场景做回归矩阵。

## 5. 场景化 Spec 拆分建议

| 优先级 | Spec | 目标 |
|---|---|---|
| P0 | 学员找老师到预约上课闭环 | 学员能从发现教师到预约、进入课堂、评价完成一节课 |
| P0 | 支付托管退款结算一致性 | 修复资金滞留和佣金统计，补支付回归测试 |
| P0 | 教师入驻排课授课收款闭环 | 教师能完成入驻、维护时段、确认授课、查看收入 |
| P1 | 课堂互动与课后评价闭环 | 从文本消息占位走向可用课堂体验和课后反馈 |
| P1 | 运营客服与争议处理闭环 | 给运营/客服处理异常交易和争议的工具 |
| P1 | 场景级回归验证体系 | 建立按用户旅程跑的回归测试与 CI |

## 6. 执行策略

本总控 Spec 不直接实现代码。后续实施按场景 Spec 执行：

1. 先完成 `01-产品规划/20260501-1120-CNVN用户场景地图`，作为所有后续 Spec 的需求源。
2. 为 P0 场景创建独立 Spec，每个 Spec 只负责一个可验收用户闭环。
3. 每个场景 Spec 内再拆前端、后端、测试、文档任务。
4. 资金相关改动必须优先做测试和一致性修复。
5. 后续代码实现仍留在当前分支 `feat/spec-20260501-1058-mvp-to-product-ready`，避免散落到多个无关分支。
