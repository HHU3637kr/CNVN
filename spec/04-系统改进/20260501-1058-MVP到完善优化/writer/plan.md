# MVP 到产品化完善：场景化 Spec 编排计划

## 1. 概述

本 Spec 原目标是把 CNVN 从 MVP 优化到较完善产品。经探索后调整为场景化编排：先整理用户使用软件的全部场景，再按场景创建独立 Spec 做优化。

原因是当前系统已经具备多个模块骨架，真正的产品化问题不是“再补几个接口”，而是每个用户角色能否完成完整任务：

- 学员能否找到老师、预约、上课、评价。
- 教师能否入驻、排课、授课、收款。
- 平台能否保证支付托管、退款、结算、争议处理正确。
- 运营能否处理异常和风险。
- 工程团队能否按场景回归验证。

## 2. 需求分析

### 2.1 用户角色

| 角色 | 核心目标 |
|---|---|
| 访客 | 理解平台价值，找到可信教师 |
| 学员 | 用越南语辅助找到中文老师，完成预约、上课和评价 |
| 教师 | 上架服务、管理时间、完成授课、获得结算 |
| 运营/客服 | 处理争议、订单异常、教师审核和用户问题 |
| 财务/风控 | 确保托管、退款、结算、出款和账本一致 |

## 3. 设计方案

### 3.1 场景优先级

P0 先处理能形成产品可信闭环的三件事：

1. 学员单课闭环：从找老师到预约、课堂、评价。
2. 支付托管一致性：确保钱不会卡死，结算不会被评价状态影响。
3. 教师供给闭环：教师能入驻、排课、确认授课、看到收款。

P1 处理体验和运营：

1. 课堂互动与课后评价增强。
2. 运营客服与争议处理。
3. 场景级回归验证和 CI。

P2 处理增长、本地化和规模化：

1. 越南语 i18n。
2. 真实 VietQR/VNPay/MoMo。
3. 移动端/PWA。
4. 推荐排序和增长转化。

### 3.2 Spec 之间的依赖

```text
用户场景地图
  ├─ 学员找老师到预约上课闭环
  ├─ 教师入驻排课授课收款闭环
  ├─ 支付托管退款结算一致性
  ├─ 课堂互动与课后评价闭环
  ├─ 运营客服与争议处理闭环
  └─ 场景级回归验证体系
```

支付一致性是所有真实交易场景的底座；学员闭环和教师闭环是第一批可交付场景；运营闭环和验证体系为后续真实用户阶段做准备。

## 4. 执行模式

`execution_mode: single-agent` 按当前 Spec Write v2.0 约定保留，表示每个具体 Spec 的实现阶段可由一个 executor 串行执行。项目级协作仍使用多个子 Agent：产品场景、前端体验、后端平台、质量验证可以并行写入各自负责的 Spec。

用户已明确不设置阶段门禁，因此文档生成、拆分和后续实现由 TeamLead 自主推进；但所有关键决策必须落盘到对应 Spec。

## 5. 实现步骤

1. 更新当前总控 Spec，使其明确转为场景化编排。
2. 创建产品规划 Spec：`CNVN用户场景地图`。
3. 创建场景级 Spec 队列，但不在规划分支直接实现。
4. 每个场景 Spec 写清现有能力、缺口、P0/P1/P2 范围、前后端设计和风险。
5. 提交当前规划分支，形成后续实现的路线图基线。
6. 按上方队列逐个执行 `$spec-start`，每个 Spec 单独创建实现分支。
7. 每个 Spec 独立完成实现、测试、review、summary 和提交。

## 6. 风险和依赖

- 如果直接实现所有场景，范围会失控；必须按场景逐个闭环、逐分支交付。
- 支付一致性缺陷会影响所有交易场景，应优先修复。
- 真实支付、视频课堂、运营后台都属于 P1/P2，不应阻塞单课闭环，但需要在路线图中明确。
- 当前仓库默认分支是 `master`，本轮工作分支为 `feat/spec-20260501-1058-mvp-to-product-ready`。
- 用户已有本地 Cursor 配置变更已被单独 stash，不纳入本分支。
