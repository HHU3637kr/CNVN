# 测试计划：场景化 Spec 拆分验证

## 1. 验证目标

本测试计划只验证“场景化拆分是否完整、可执行、可追踪”，不验证业务代码实现。

后续每个场景 Spec 进入实现时，需要在自己的目录下生成更具体的测试计划和测试报告。

## 3. 内容质量检查

每个场景 Spec 必须包含：

- 用户目标。
- 现有页面/API/服务。
- 当前缺口。
- P0/P1/P2 范围。
- 前端、后端、数据/状态、异常路径设计。
- 实现步骤。
- 风险和依赖。
- 与当前总控 Spec 和场景地图 Spec 的关联。

## 4. 优先级一致性检查

P0 必须覆盖：

1. 学员单课闭环。
2. 教师供给和授课闭环。
3. 支付托管、退款、结算一致性。

P1 可以覆盖：

1. 课堂互动增强。
2. 运营/客服/争议处理。
3. 场景级回归验证体系。

P2 保留为路线图，不在本轮强行实现。

## 5. 后续实现验收原则

当某个场景 Spec 进入实现时，必须按用户旅程验收，而不是只按接口或页面验收。例如“学员找老师到预约上课闭环”必须验证：

1. 访客能搜索并进入教师详情。
2. 登录学员能选择时段并创建课程。
3. 教师能确认课程。
4. 双方能进入课堂。
5. 课程结束后学员能评价。
6. 异常路径有明确提示。
