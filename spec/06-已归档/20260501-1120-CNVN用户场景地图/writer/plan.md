# CNVN 用户场景地图

> [!important] 总控边界
> 本 Spec 只做产品场景拆解、优先级判断和后续 Spec 索引，不开始任何代码实现，不修改前端、后端、数据库或测试文件。

## 1. 概述

### 1.1 背景

CNVN（中越通）是面向越南市场的中文学习双边撮合平台，连接想学中文的越南人与兼职中文老师。当前项目已具备前后端基础闭环：认证、教师列表与详情、钱包/支付订单、预约、课堂文字实时消息、评价、教师结算/出款等能力已有不同程度落地。

随着基础能力增多，后续开发需要从“模块实现”转为“按真实用户场景补齐闭环”。本 Spec 作为产品规划层的场景地图，统一说明用户全旅程、核心角色、当前缺口和后续 Spec 拆分顺序。

### 1.2 目标

1. 梳理访客、学员、教师、运营/客服、平台财务/风控在 CNVN 中的核心使用场景。
2. 将“发现平台 → 付费预约 → 上课 → 评价/退款/结算 → 复购”的全旅程拆成可执行的后续 Spec。
3. 标注现有页面/API 与当前缺口，避免重复建设，也避免只做孤立页面。
4. 给出 MVP 到产品化路线：先单课闭环，再交易安全，再教师供给，再运营风控，再增长本地化。

### 1.3 范围

**包含**：

- 产品定位与角色定义。
- 用户全旅程场景地图。
- 场景清单表：场景编号、角色、触发条件、用户目标、现有页面/API、当前缺口、优先级、建议后续 Spec。
- MVP 到产品化路线。

**不包含**：

- 不实现代码。
- 不设计数据库迁移。
- 不新增 API 契约细节。
- 不替代后续 `03-能力交付`、`04-系统改进`、`05-验证工程` 中的具体实现 Spec。

## 2. 产品定位与角色

### 2.1 产品定位

CNVN 是“越南人学中文”的双边撮合与交易平台。平台价值不是单纯提供课程内容，而是把学生找老师、老师接单、平台担保交易、课堂履约、评价沉淀和结算出款组织成一个可信闭环。

### 2.2 核心角色

| 角色 | 定义 | 核心诉求 | 平台责任 |
|---|---|---|---|
| 访客 | 未登录用户，可能来自搜索、社交传播、转介绍 | 快速判断平台是否可信、是否有合适老师 | 展示老师供给、价格、评价、上课方式与注册入口 |
| 学员 | 想学习中文的越南用户 | 找到合适老师、低成本试课、方便支付、按时上课、遇到问题可退款/申诉 | 提供搜索、详情、充值/支付、预约、课堂、评价、退款争议入口 |
| 教师 | 在华越南留学生、越南本地中文导游、在越中国人、专业中文老师等供给方 | 展示能力、设置时段、确认课程、稳定收款、积累评价 | 提供档案、可用时段、课程管理、课堂、收入结算、出款记录 |
| 运营/客服 | 负责人工审核、异常处理、用户支持 | 看见异常、处理充值凭证/退款争议/教师违规、维护体验 | 提供后台工单、订单查询、证据链、手工处理工具 |
| 平台财务/风控 | 负责资金安全、对账、出款、支付风险控制 | 资金账实一致、避免重复付款/重复退款、识别异常交易 | 提供支付订单、钱包流水、结算快照、出款审核、风控规则 |

## 3. 用户全旅程

### 3.1 主链路

1. **发现平台**：访客通过搜索、社交媒体、老师分享或朋友推荐进入首页/老师列表。
2. **注册登录**：访客注册为学员；潜在供给方注册后开通教师身份。
3. **找老师**：学员按价格、评分、教师类型、擅长方向、可用时间筛选老师。
4. **看详情**：学员查看教师介绍、价格、评价、可用时段和是否适合自己的学习目标。
5. **充值/支付**：学员通过钱包 Mock 充值、VietQR/本地转账占位或支付订单完成资金准备。
6. **预约**：学员选择老师和时间，系统校验可用时段、冲突和余额，生成待确认课程。
7. **教师确认**：教师在课程中心确认或拒绝待确认课程。
8. **课前准备**：师生查看课程时间、课堂入口、消息历史和课件/备注占位。
9. **进入课堂**：课程开始前后进入课堂页。
10. **课堂消息/视频占位**：当前课堂支持 WebSocket 文字实时消息，音视频为界面占位。
11. **结束课程**：师生结束课程，课程进入 completed，并触发结算逻辑。
12. **评价**：学员对已完成课程评价，评价影响教师评分和后续排序。
13. **退款/争议**：取消、未确认过期、支付异常、课堂异常或服务不满意时进入退款/争议处理。
14. **教师结算/出款**：课程完成后按平台费率结算，教师查看收入与出款单。
15. **运营处理异常**：运营/客服处理线下转账凭证、支付订单异常、争议、退款、违规和用户投诉。
16. **复购**：学员基于上课体验再次预约同一老师，或继续搜索新老师。

### 3.2 场景分层

| 层级 | 场景目标 | 判断标准 |
|---|---|---|
| 单课闭环 | 学员能从找老师到完成一节课并评价 | 一节课可完成支付、预约、确认、课堂、结束、评价 |
| 交易安全 | 资金状态、退款规则、结算出款可解释且可追溯 | 钱包流水、支付订单、结算快照、争议处理不互相冲突 |
| 教师供给 | 教师愿意长期维护档案和时段 | 教师能看见收益、待确认课程、评价反馈和出款进度 |
| 运营风控 | 异常不依赖开发查库处理 | 运营有后台视图、处理动作和审计记录 |
| 增长本地化 | 越南用户能理解、信任并转化 | 越南语、VND、VietQR/MoMo/ZaloPay、本地化 SEO/分享链路 |

## 5. 优先级判断

### 5.1 P0：单课交易闭环必须完整

P0 不等于“所有功能都做满”，而是任何一个断点都会让平台无法验证商业闭环。优先补齐：

1. 注册登录与角色引导。
2. 找老师、看详情、下单预约。
3. 充值/支付确认。
4. 教师确认。
5. 课堂进入、开始、结束。
6. 课后评价。
7. 退款/争议和运营客服最小处理。

### 5.2 P1：让交易安全、供给稳定

P1 聚焦可规模化运营：

1. 课前准备与提醒。
2. 教师收入与结算明细。
3. 教师出款申请与财务审核。
4. 支付对账与风控审计。
5. 教师入驻与供给质量。

### 5.3 P2：增长与本地化

P2 聚焦越南市场增长效率：

1. 复购与学习记录。
2. 越南语 i18n 与本地化体验。
3. 场景漏斗指标、邀请、分享与增长验证。

## 6. MVP 到产品化路线

### 6.1 阶段一：先单课闭环

目标：一个学员可以完成“一次找老师、一次付费、一次预约、一次上课、一次评价”。

建议拆 Spec：

- `03-能力交付/注册登录与角色引导闭环`
- `03-能力交付/找老师筛选与排序体验`
- `03-能力交付/教师详情到预约转化`
- `03-能力交付/单课预约下单体验`
- `03-能力交付/课堂单课履约闭环`
- `03-能力交付/课后评价与教师口碑沉淀`

### 6.2 阶段二：再交易安全

目标：资金状态可追踪，退款/争议有入口，运营可以不查库处理异常。

建议拆 Spec：

- `03-能力交付/越南本地支付与充值确认闭环`
- `04-系统改进/退款争议与客服处理链路`
- `04-系统改进/运营客服后台最小闭环`
- `04-系统改进/支付对账与风控审计`

### 6.3 阶段三：再教师供给

目标：教师愿意长期维护档案和时段，知道自己的收入与出款状态。

建议拆 Spec：

- `03-能力交付/教师确认与响应率机制`
- `03-能力交付/教师入驻与供给质量`
- `03-能力交付/教师收入与结算明细`
- `03-能力交付/教师出款申请与财务审核`

### 6.4 阶段四：再运营风控

目标：客服、财务、风控有统一处理台和审计链路，平台能承接真实交易异常。

建议拆 Spec：

- `04-系统改进/运营客服后台最小闭环`
- `04-系统改进/支付对账与风控审计`
- `05-验证工程/关键交易状态机回归测试`
- `05-验证工程/退款争议人工处理验收`

### 6.5 阶段五：再增长本地化

## 9. 风险与依赖

| 风险 | 影响 | 应对 |
|---|---|---|
| 只补页面不补状态机 | 用户看得到入口但交易无法完成 | 后续 Spec 必须按场景写验收链路 |
| 支付/钱包/订单/结算概念混杂 | 退款和出款容易出现重复或账实不一致 | 交易安全阶段必须统一订单、流水、结算快照和审计口径 |
| 教师供给不足 | 学员即使注册也无法找到合适老师 | 在单课闭环后尽快推进教师入驻与供给质量 |
| 运营后台滞后 | 一旦真实用户付款，异常需要人工查库 | P0 保留运营客服最小闭环 |
| 越南本地化不足 | 首访转化和支付信任低 | 产品化阶段推进越南语、VND、支付方式和本地客服渠道 |
