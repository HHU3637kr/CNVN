# 支付模块结算防重与数据模型关联

## 概述

在实现 CNVN 支付模块的教师结算功能时，遇到了两个关键问题：防重复结算和 Lesson 模型与 User 模型的关联关系。

---

## 经验 1：防重复结算机制

### 困境

课程完成时触发结算，但可能因以下原因导致重复结算：
- 网络重试导致 `end_lesson` API 被多次调用
- 并发请求同时处理同一课程
- 系统故障恢复后重复执行结算逻辑

### 策略

**数据库状态检查 + 事务原子性**：

```python
async def settle_teacher_lesson(db: AsyncSession, lesson: Lesson) -> None:
    # 1. 防重复结算检查
    if lesson.settled_at is not None:
        return  # 已结算，幂等返回

    # 2. 计算费率和金额...
    # 3. 执行入账操作...

    # 4. 标记课程已结算
    lesson.settled_at = datetime.now(timezone.utc)
    lesson.teacher_amount = teacher_amount
    lesson.platform_fee_rate = Decimal(str(fee_rate))

    await db.commit()
```

**要点**：
1. 在 `Lesson` 模型增加 `settled_at: datetime | None` 字段作为结算标记
2. 结算前先检查该字段，如已设置则直接返回（幂等性）
3. 使用数据库事务保证整个结算流程的原子性
4. 结算完成后立即设置标记

### 理由

- **轻量级**：不需要引入分布式锁，利用数据库状态即可
- **幂等性**：重复调用不会产生副作用
- **可追溯**：`settled_at` 记录了结算时间，便于审计

## 经验 2：Lesson.teacher_id 与 User.id 的关联关系

### 困境

结算时需要操作教师钱包，但 `Lesson.teacher_id` 指向的不是 `User.id`，而是 `TeacherProfile.id`。直接用 `teacher_id` 查询钱包会找不到记录。

### 策略

**通过 TeacherProfile 中转关联**：

```python
# Lesson.teacher_id 是 TeacherProfile.id
# 需要先查询 TeacherProfile 获取 user_id

from app.models.teacher_profile import TeacherProfile

r = await db.execute(
    select(TeacherProfile).where(TeacherProfile.id == lesson.teacher_id)
)
teacher_profile = r.scalars().first()
if not teacher_profile:
    raise ValueError(f"教师档案不存在: {lesson.teacher_id}")

teacher_user_id = teacher_profile.user_id

# 然后用 teacher_user_id 操作钱包
await wallet_service.credit_settlement(
    db, teacher_user_id, teacher_amount, lesson.id, ...
)
```

**数据模型关系**：
```
User (id, ...)
  └── 1:1 ── TeacherProfile (id, user_id, ...)
                    ↑
                    │ lesson.teacher_id 引用这里
                  Lesson
```

### 理由

- CNVN 的数据模型设计是 `User → TeacherProfile (1:1)`
- `Lesson` 引用的是 `TeacherProfile.id`，便于直接获取教师公开信息
- 钱包属于 `User`，所以需要通过 Profile 获取 user_id

---

## 适用场景

- **防重机制**：涉及资金操作的批量/定时任务、可能重试的 API 端点
- **模型关联**：需要通过教师 ID 获取用户信息（钱包、权限等）的场景
